# 测试用例质量与统一格式

## 风险覆盖模型

生成 TC 前必须建立：需求点 → 风险 → 条件组合 → 数据形态 → 业务入口 → 预期差异 → TC。模板和 Profile 只用于扫描，未命中证据的项目不得生成 TC。

需求明确列出两个及以上条件维度时，交接顺序固定为：原始需求 → Evidence Snapshot → Fact/Requirement → 条件矩阵 → Risk Matrix → Testcase Model → XMind。条件维度包括业务入口、权限/角色/关系类型、目标范围、状态、操作、数据形态、命中结果和 AND/OR 连接方式。条件矩阵必须记录稳定维度 ID、枚举值、分组笛卡尔积生成契约、必测组合、排除组合、排除原因和覆盖 TC；生成契约的每个分组明确固定值、可变维度及预期组合数，程序生成的 expected set 必须与 required + excluded 的规范化完整维度键完全一致。每个明确枚举值及 required combination 都必须有行为覆盖，未覆盖为 error。排除组合必须有可审计依据，不得以“其他组合同理”跳过明确枚举，也不得无证据机械生成无意义全局笛卡尔积。

权限、金额、状态流转和数据隔离属于高风险决策矩阵；等价类或两两组合压缩必须记录等价依据。权限需求明确列出“权限类型 × 关系 × 目标范围”时，每个组合至少包含命中与不命中数据以及可见/不可见结果。普通人员的等于、不等于、包含于、不包含、完全包含于与自己、所有下级、直属上级、直属下级必须逐项表达；允许把四种目标范围组织在同一“角色+关系”TC 内，但不得隐藏为“分别验证全部目标”。指定用户关系必须分别覆盖单用户、多用户、命中和全部不命中；没有 Evidence 时不得扩写重复用户去重、拒绝选择、提示或删除后保存行为。

配置存在性与配置行为必须分离：控件、关系选项、目标选项或多用户追加只证明结构/容量，不计入 required combination 的行为覆盖。行为型覆盖必须写明权限类型、关系、目标范围或用户集合、命中/不命中数据及可观察业务结果。

每条 TC 必须回答：

1. 验证哪个需求、Diff 或历史缺陷。
2. 验证哪个核心风险。
3. 为什么必须独立存在。
4. 失败后能定位什么差异。

无法回答时删除或合并。每条 TC 只保留一个业务测试点分支，避免一条用例混合多个不可独立定位的风险。

## 最小有效用例集和去重

兼容 `deduplication_key` 可继续包含业务入口；合并判断使用确定性的 `core_deduplication_key`。核心键至少由业务对象、触发条件、核心动作、核心断言和风险语义组成，不包含纯入口名称；数据源、权限规则、计算口径和异常处理可作为防止误合并的补充因子。

以下差异优先合并：

- 正式与模拟名称不同但规则相同。
- 多个页面、弹窗或模块入口名称不同但数据来源、步骤和预期相同。
- 字段名不同但必填、格式、长度或范围规则完全相同。
- 数据值不同但属于同一等价类。
- 步骤和核心断言相同。

只有核心规则、触发方式、操作、数据来源/口径、预期、权限、参数、异常处理、风险或失败定位不同才拆分。公共入口覆盖范围写在入口节点或步骤中。同一核心规则覆盖多个真实入口时仍保留一个 TC，但必须把入口拆成独立、平级、可独立执行和定位的分支；入口差异本身不是拆 TC 的理由。

两个 TC 的核心键相同且只存在模拟/正式、页面、弹窗或模块入口名称差异时必须合并为一个 TC 的多个 `entry_branches`。只有数据源、权限规则、计算口径、操作、Oracle、异常处理或无法由分支独立定位的风险确实不同才允许拆分，并记录拆分原因。Manifest 的 `case_count` 只统计 TC，`branch_count` 单独统计真实入口分支。

## 测试方法

按风险选择等价类、边界值、决策表、状态迁移、用户路径、因果图、正交或两两组合。新增能力覆盖主链路和真实异常；既有逻辑变更聚焦变更条件、反向场景、临界值和受影响调用方。

必须扫描但仅在证据命中时生成：空值、零值、负值、极值、重复、顺序、精度、时区、权限、并发、超时、降级、迁移、兼容和大数据量。

## 禁止模糊和虚构断言

“页面正常”“功能正常”“查询正常”“展示正常”“数据正常”等无对象、无状态值、无变化条件的表达必须改写。明确枚举或可观察状态流转不属于模糊断言，例如“任务状态由异常变更为正常”“状态字段值为 NORMAL”“设备状态显示‘正常’”。

“按已确认规则处理”“按系统现有逻辑处理”“按规则处理”“处理成功”“展示正确”同样属于不可观察的模糊断言。“按关系判断”“按关系得到可观察结果”“根据关系判断可见性”“根据对应关系处理”“关系生效”“权限结果正确”“按配置结果展示”统一报 `VAGUE_RELATION_ORACLE`；必须改写为明确输入条件及可见/不可见结果。字段存在或支持多值仅能生成结构/容量用例，不能自动生成过滤、统计排除、去重、保存、删除或继承行为用例。

“按现有统计口径一致”“按现有口径处理”“保持现有逻辑”必须给出计算对象、汇总来源及可比对字段或 Oracle；“未发生本需求之外的变化”“其他功能不受影响”“与原来一致”“没有其他影响”“保持不变”“不受影响”必须给出变更前基线、同条件比较对象及至少一个可观察字段或结构。明确写出同账号/同查询条件的变更前快照及逐字段对比，或可见行逐字段求和与汇总行对应字段对比，允许通过。

完全一致或仅业务对象名称不同且规则相同的重复用例作为 error；文本相似但无法确定等价时作为 warning。正式与模拟数据源、权限范围、金额与日期类型、异常处理和不同 P0 风险存在差异时不得自动合并。warning 默认不阻止发布，`--strict` 可将 warning 升级为门禁。

预期必须明确可观察对象和判定条件，例如请求参数、结果集合、状态变化、精度和舍入、顺序、权限响应、错误码或持久化结果。

未知枚举、状态和兜底逻辑不得写死不展示、展示为空、默认过滤或默认值。把未知内容记录为待确认点；非阻塞时只验证没有未处理异常，具体业务结果以确认口径为准。不得把当前代码行为直接当成业务预期。

SQL、接口和字段只能使用证据中存在的标识；禁止某表、某字段、xxx、假设表等占位事实。

## 测试用例价值评估

### 评估目标

测试用例价值评估（Testcase Value Assessment）是 Testcase、Risk、Requirement 和 Evidence 的派生评估结果，用于识别高价值核心用例、长期回归保留用例、低价值 reachability 冒烟、疑似重复、多风险混合导致的诊断价值不足、低风险高维护成本，以及高优先级但证据不足的用例。它不是 `test_priority`、`business_impact`、执行结果、覆盖率或人工工作量估算的替代品，也不能作为修改业务代码的事实依据。

### 七维评分

评估维度固定为：

- `business_impact`
- `risk_coverage_value`
- `regression_value`
- `diagnostic_value`
- `evidence_confidence`
- `maintenance_cost`
- `redundancy_penalty`

七个维度和总分均由程序确定性计算，用户或 AI 不得直接填写分数。权重方向强调业务影响、风险覆盖、回归复用、失败诊断和证据可信度，并扣减维护成本和重复惩罚；唯一评分公式、映射和取整规则只在 `scripts/qa_contracts.py` 维护，规则文档不复制实现代码。

### 保护规则

- 映射 P0 Risk 的 TC 必须保留；`historical_defect_ids` 非空的回归 TC 也必须保留。
- 受保护用例低分时只能提出改善建议，不得自动删除、自动合并、自动降级或自动跳过。
- 疑似重复只触发人工复核，不得据此自动改变 TC。
- 测试用例价值评估不能覆盖 P0/P1/P2、风险映射、证据状态或回归范围。

### 阶段一边界

- Testcase Value Assessment Model 是独立、可选、非阻塞模型；未生成 Assessment 不影响现有正式产物。
- warning 和 suggestion 仅供人工审核，不阻塞交付；只有结构、引用、Hash 或持久化重算结果错误才作为 error。
- Assessment 不写入 Testcase Model 或 XMind Markdown，不参与 Manifest 计数，也不改变 TC 数量、分支数或执行实例数。
- Assessment 独立于 Execution Model，不代表测试已经执行。
- `score_status=insufficient_inputs` 只输出输入不足原因和固定 recommendation，不得根据全零或不完整维度产生低价值、简化、重复、拆分或其他依赖总分/价值带的建议。

## 固定 XMind Markdown

存在公共入口：

    - 业务根节点
        - 功能测试
            - 公共入口或业务对象
                - TC001
                    - 业务测试点
                        - 操作步骤
                            - 预期结果

同一测试点覆盖多个入口时：

    - 业务根节点
        - 功能测试
            - 公共入口或业务对象
                - TC001
                    - 业务测试点
                        - 具体入口一
                            - 操作步骤
                                - 预期结果
                        - 具体入口二
                            - 操作步骤
                                - 预期结果

不存在公共入口：

    - 业务根节点
        - 功能测试
            - TC001
                - 一级业务模块
                    - 二级功能点
                        - 操作步骤
                            - 预期结果

无公共入口的多入口用例使用：TC、一级业务模块、二级功能点、具体入口、步骤、预期；入口仍必须是二级功能点下的至少两个平级分支。

固定维度仅允许：功能测试、数据测试、异常测试、权限测试、导出测试、兼容性测试、回归测试、SQL验证。

## 格式强制约束

- 只有一个具备业务语义的根节点，禁止命名为测试用例、用例或 Test Cases。
- 每个一级节点必须是固定维度。
- 同一维度选择一种结构：TC 直接位于维度下，或 TC 位于一个公共入口下；不得混用或额外插层。
- 无公共入口时 TC 到叶节点严格为 TC、一级模块、二级功能点、步骤、预期五层。
- 有公共入口时 TC 到叶节点严格为 TC、测试点、步骤、预期四层。
- 有公共入口的多入口 TC 严格为 TC、测试点、具体入口、步骤、预期；无公共入口的多入口 TC 严格为 TC、一级模块、二级功能点、具体入口、步骤、预期。
- 单入口不得人为增加具体入口层；多入口至少两个平级入口，每个入口独立包含步骤和预期，禁止直接步骤与入口分支混用。
- 入口必须是真实业务名称，禁止“入口A”“页面1”等占位符；禁止把“分别打开/依次进入多个入口”拼接在同一步骤中。普通组合操作（例如输入股票代码和交易日期）不视为多入口。
- 每个 TC 仍只有一个核心测试点分支；步骤和预期使用短文本。
- TC 从 TC001 开始，在所有维度中全局连续且不重复，TC 节点只包含编号；按主维度分组后不要求树的遍历顺序仍按编号递增。
- 每级缩进使用 4 空格，禁止 Tab 和层级跳跃，只允许短横线列表。
- 本地文件禁止代码块、表格、JSON、标题、说明文字、总结、空节点和标签式节点。
- 标签式节点包括模块、测试点、操作步骤、预期结果及其全角或半角冒号形式。
- 对话中只有用户明确索要完整 Markdown 时才使用代码块展示。

## XMind 节点语义精简

- 不设置测试点、入口、步骤或预期的固定字符上限；不得仅因文本较长输出 error 或 warning。
- 不得自动截断、添加省略号，或删除业务对象、核心条件、操作和可观察预期。无法安全精简时保留完整表达，准确性优先于简短。
- 优先删除重复背景、说明性套话以及父节点已经表达的页面、弹窗、系统或业务对象名称；同一节点原则上表达一个主要动作或一个主要可观察结果。
- XMind 只保留执行所需的业务对象、核心条件、操作和可观察预期。详细背景、字段说明、来源证据、权限矩阵、评分和内部 Requirement/Risk/Evidence 字段保留在结构化模型，不重复塞入 XMind。
- 多个可独立失败的业务断言应拆分；不得只为缩短文字强行拼接，也不得改变既有单入口和多入口固定层级、TC 编号或 TC 数量。
- 语义明确且无歧义时可使用 `=`、`≠`、`>`、`<`、`≥`、`≤`、`∈`、`∉`、`AND`、`OR`、`→`、`+`、`-`、集合和枚举表达，例如 `role_type=30 AND enabled_flag=1`、`role_type∈{10,20,30}`、`未配置→已配置`。
- `包含于`、`完全包含于`、`任一包含`、`全部包含`等集合口径未确认时必须保留明确文字；字段名、枚举值或状态值没有真实证据时不得用符号伪造确定规则。
- 同时使用 `AND` 与 `OR` 且优先级可能有歧义时必须使用括号，例如 `(A AND B) OR C`；不得写成无法确定优先级的 `A AND B OR C`。
- 使用符号后仍必须能识别动作主体、业务对象和可观察结果；校验器只识别截断、歧义和既有结构问题，不自动改写文本。
