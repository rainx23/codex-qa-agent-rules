# 分析报告契约

本文件是分析报告模式、章节和校验要求的唯一正文来源。Skills、README 和兼容文档只引用本文件，不复制完整规则。

## 通用要求

- 报告可使用 `报告模式：纯需求`、`报告模式：纯Diff` 或 `报告模式：需求与Diff联动` 显式声明模式。
- 校验命令可使用 `--mode requirement`、`--mode diff` 或 `--mode combined` 指定模式；未指定时先读取报告模式字段，再按章节和证据自动识别。
- 章节标题允许无编号、中文数字编号或阿拉伯数字编号，以及括号、顿号、中英文句号和空格分隔形式。
- 章节存在判断、正文提取、模式识别、证据、P0、疑似缺陷和追踪矩阵校验必须使用同一份标准化章节模型。
- 所有必需章节必须有正文；“无”也需要明确写出结论。
- 正式测试用例任务必须增加“测试维度扫描”表，固定展示八个测试分类维度的状态、主要风险、对应 TC 与原因或阻塞；没有 TC 的维度也不得省略。
- 需求涉及数据或指标时，报告应增加“历史知识命中、数据影响分析、数据验证结论、指标口径、验证 SQL 计划、直接对数方案、DDL 或字段补充需求、知识沉淀草稿”章节；无数据影响时可写明结论和原因。
- 指标准确性默认需要 SQL 计划；只有用户或明确验收文档提供独立可信的直接对数关系时，才允许使用 `cross_source_reconciliation`。

## 纯需求报告

适用于只有禅道、OpenSpec、Markdown、截图或用户补充，没有实际 Diff 证据的分析。

必须包含：本次分析范围、需求理解、规则拆解、证据来源、待确认点、风险点、测试点摘要、回归范围。

可以包含验收基准、业务目标、不在范围、默认假设和疑似缺陷。纯需求报告不强制输出疑似缺陷或追踪矩阵；主动输出疑似缺陷时，不得把需求歧义或规则缺失直接认定为代码缺陷。

## 纯 Diff 报告

适用于只有 Commit、Diff、分支、工作区或暂存区变更，没有明确需求验收基准的分析。

必须包含：本次分析范围、Commit 或 Diff 对比范围、Diff 涉及文件、核心改动点、证据来源、待确认点、疑似风险点、疑似缺陷、测试点摘要、回归范围。

疑似缺陷章节必须明确结论。没有需求基准或双重证据时，说明本次未发现具备需求证据和 Diff 证据的疑似缺陷；实现层面的不确定影响写入疑似风险点。没有需求基准时不强制追踪矩阵。

## 需求与 Diff 联动报告

适用于需求作为实现验收基准，并检查 Diff 是否覆盖、遗漏或偏离需求的分析。

必须包含：本次分析范围、需求理解、Diff 理解、证据来源、待确认点、风险点、疑似缺陷、测试点摘要、回归范围、需求-Diff-测试点追踪矩阵。

追踪矩阵的正式字段、行级映射和覆盖状态要求以 `rules/core/traceability-rules.md` 为唯一正文来源。

疑似缺陷结论必须包含需求证据、Diff 证据、证据状态和影响。缺少任一项时只能称为疑似风险，不得称为疑似缺陷。

## 模式识别优先级

1. 命令行 `--mode`。
2. 报告中的显式“报告模式”字段。
3. 标准化后的章节结构和证据来源。

自动识别时，追踪矩阵或需求与 Diff 章节同时存在视为联动；只有需求理解视为纯需求；出现 Diff 理解、核心改动点、Commit/Diff 范围或变更文件视为纯 Diff。无法稳定判断时要求显式指定，不凭模糊正文猜测。
