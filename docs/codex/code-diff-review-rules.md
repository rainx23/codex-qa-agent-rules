# 代码 Diff 测试分析规则

## 适用场景

当用户提到以下内容时，按本规则执行：

- 代码 diff
- commit 分析
- gray 单分支提测
- 变更评审
- 回归范围判断
- 基于 diff 生成测试点和 XMind 用例

## 默认角色和边界

- 默认以资深测试专家、测试架构师身份分析。
- 默认只做测试分析，不修改任何业务代码。
- 除非用户明确说“修改代码 / 修复代码 / 提交代码”，否则禁止改业务代码。
- 本规则重点服务于 `gray` 单分支提测场景。
- 证据不足时，只输出风险点和待确认点，不生成确定性预期，不虚构 SQL、字段、接口、页面入口。

## 输出模式规则

- 支持三种输出模式：只分析、生成用例、完整输出。
- 只分析：只生成 diff 分析报告，不生成 XMind 用例文件。
- 生成用例：只生成可直接导入 XMind 的 Markdown 用例文件；如证据不足，只生成待确认类风险说明，不生成确定性用例。
- 完整输出：同时生成 diff 分析报告和 XMind 用例文件。
- 如果用户未指定输出模式，默认使用完整输出。
- 支持两种用例范围：只列 P0、完整用例。
- 只列 P0：仅输出 P0 风险、P0 测试点和 P0 XMind 用例。
- 完整用例：覆盖 P0/P1/P2 关键场景，但禁止低价值、重复或脱离业务的泛化用例。
- P0/P1/P2 判定统一参考 `docs/codex/qa-priority-rules.md`。
- 常见业务类型的影响分析可参考 `docs/codex/qa-common-templates.md`，但不得把模板机械转成用例。

## 单 Commit 分析规则

- 如果用户只提供一个最新 commit，默认分析该 commit 相对上一个父提交的改动。
- 推荐命令：
  - `git show <commit>`
  - `git diff <commit>^..<commit>`
- 分析前必须校验 commit 是否存在。
- 如果 commit 不存在，必须说明，不允许输出伪造测试点。
- 如果 commit 是 merge commit，需要说明它可能有多个父提交，并提示确认父提交或对比范围。
- 如果用户明确要求继续分析 merge commit，可按第一父提交进行初步分析，并在报告中标注该假设。

## 双 Commit 范围分析规则

- 如果用户提供两个 commit，严格按用户给定的旧版本和新版本对比：
  - `git diff <old_commit>..<new_commit>`
- 如果用户提供完整表达式，如 `old_commit..new_commit` 或 `old_commit...new_commit`，按原表达式执行。
- 分析前分别校验 old commit 和 new commit 是否存在。
- 不允许擅自替换用户提供的 commit。
- 如果 diff 为空，必须说明可能原因，不允许编造测试点。

## Gray 单分支提测规则

- 当前项目提测通常直接在 `gray` 分支。
- 没有独立 feature 分支时，不强制要求开发分支和目标分支。
- 单分支提测优先按 commit 或 commit 范围分析。
- 如用户只给最新 commit，则默认分析该 commit 本身改动。
- 如用户给两个 commit，则分析两个版本之间完整改动。
- 只有当用户明确要求切换 `gray` 并拉取最新代码时，才执行：
  - `git switch gray`
  - `git pull --ff-only origin gray`

## 需求上下文关联分析规则

- 如果当前对话或上下文中已经有需求文档、OpenSpec、禅道需求或用户手动粘贴的需求内容，分析 diff 时必须引用这些需求上下文。
- diff 分析不能只看代码变更本身，必须判断代码是否满足需求规则、是否遗漏需求点、是否实现偏差、是否引入需求外变更。
- 分析顺序必须是：
  - 先提炼需求规则和验收点。
  - 再读取 diff 文件清单和代码差异。
  - 再建立“需求规则 -> 代码变更 -> 测试点 / 风险点”的映射。
- 输出中必须包含“需求与 Diff 对照结论”，说明每个核心需求点对应的代码实现情况：已覆盖、疑似遗漏、实现不一致、无法判断。
- 如果需求与 diff 存在冲突，必须列为 P0/P1 风险或待确认点，不能直接按代码实现生成确定性预期。
- 如果需求证据不足但 diff 中出现相关实现，只能说明“代码当前实现如此”，不能反推为确定需求。
- 测试点生成优先级：需求核心规则 > 需求与 diff 不一致处 > diff 高风险实现 > 回归影响范围。

## 需求-Diff-测试点追踪矩阵

当存在需求上下文时，diff 分析报告必须输出追踪矩阵。

建议字段：

- 需求点
- 需求证据来源
- diff 覆盖情况：已覆盖、疑似遗漏、实现不一致、需求外变更、无法判断
- diff 证据：文件路径、函数、接口、SQL 或变更摘要
- 风险等级：P0/P1/P2
- 测试点或 TC 编号
- 待确认点编号

如果没有需求上下文，应说明“未提供需求基准，本次仅基于 diff 证据分析风险”，不得反推需求事实。

## 需求与 Diff 不一致输出规则

发现不一致时，必须输出：

- 不一致项：需求描述与 diff 表现的差异。
- 需求证据：需求原文、用户补充或 OpenSpec 依据。
- diff 证据：文件、接口、SQL、函数或变更摘要。
- 影响范围：页面、接口、数据、权限、导出、下游调用或回归范围。
- 风险等级：参考 `docs/codex/qa-priority-rules.md`。
- 建议处理：产品确认、研发确认、补实现、补回归或记录风险。
- 用例处理：生成风险验证或通用验证，不把代码当前实现写成确定需求预期。

## 固定 Git 校验流程

分析 diff 前，优先执行或按需执行以下只读命令：

- `git status --short`
- `git rev-parse --abbrev-ref HEAD`
- `git rev-parse --verify <commit>^{commit}`
- `git rev-list --parents -n 1 <commit>`
- `git diff --name-status <old>..<new>`
- `git diff --stat <old>..<new>`
- `git diff <old>..<new>`

说明：

- `git rev-parse --verify` 用于确认 commit 是否存在。
- `git rev-list --parents` 用于判断是否 merge commit。
- `git diff --name-status` 用于获取变更文件清单。
- `git diff --stat` 用于了解整体改动规模。
- `git diff` 用于读取具体代码差异。

## Diff 文件清单分析规则

分析 diff 时必须输出变更文件清单，并区分：

- 新增文件
- 修改文件
- 删除文件
- 重命名文件
- SQL 文件
- Groovy / 接口编排文件
- 配置或公共逻辑文件
- 测试文档或非业务文件

对每个关键文件说明：

- 所属模块
- 变更类型
- 可能影响的页面、接口、报表、查询、导出或定时任务
- 是否属于公共逻辑
- 是否可能影响多个调用方

## Diff 变更类型分类规则

分析关键文件时，应识别以下变更类型：

- 页面展示变更：字段、列、按钮、弹窗、跳转入口、+N 展开、文案。
- 查询条件变更：默认值、筛选条件、多选、日期、枚举、条件联动。
- 接口契约变更：入参、出参、返回结构、错误码、默认参数、兼容旧参数。
- SQL 和数据口径变更：where、join、group by、聚合、去重、排序、分页。
- 权限和数据范围变更：角色、组织、部门、数据隔离、导出权限、越权拦截。
- 状态和流程变更：状态机、审批、终态、异常状态、重复提交。
- 配置和开关变更：配置默认值、灰度范围、生效时机、回滚。
- 公共逻辑变更：公共组件、公共接口、公共 SQL、公共工具方法。
- 非业务变更：文档、注释、格式化、日志文案、无业务含义重命名。

每类变更都应输出可能影响的测试方向；非业务变更不强行生成业务用例。

## 调用链和影响链分析规则

- 对接口、公共方法、公共 SQL、公共组件、配置项变更，必须追踪直接调用方和可能的间接影响。
- 优先使用 `rg` 搜索调用引用、接口路径、方法名、SQL id、配置 key、字段名。
- 影响链至少说明：变更点 -> 直接调用方 -> 受影响页面/接口/任务 -> 测试关注点。
- 如果调用链无法确认，必须说明无法确认原因，并列为风险或待确认点。
- 不允许在未找到证据时虚构调用方。

## 接口契约变更分析规则

出现接口入参、出参、错误码、路径、请求方法、返回结构、默认值或兼容逻辑变化时，必须分析：

- 新旧调用方是否兼容。
- 必填参数、可选参数、空值、多值、类型错误处理。
- 返回字段新增、删除、重命名、类型变更是否影响前端解析。
- 错误码和异常返回是否被调用方处理。
- 接口直接访问是否受权限和数据范围控制。
- 是否需要覆盖旧参数、旧入口、历史链接或缓存数据。

## 安全影响分析规则

出现权限、组织、角色、用户、数据范围、导出、下钻、接口参数、SQL 条件变化时，必须分析安全影响：

- 是否可能越权查看或操作。
- 是否可能跨组织、跨部门、跨角色访问数据。
- 是否可能通过参数篡改绕过筛选或权限。
- 导出、弹窗、下钻、接口直接访问是否继承权限。
- 敏感数据是否新增展示、导出或返回。

安全风险证据不足时，列为风险点或待确认点，不生成确定性结论。

## 性能影响分析规则

出现 SQL、分页、排序、导出、聚合、大数据量查询、循环调用、批量处理、外部接口调用变化时，必须分析性能影响：

- 是否新增大范围查询、全表扫描、低选择性条件或复杂 join。
- 是否改变分页、排序、聚合、distinct、group by 口径。
- 是否新增循环内查询、重复接口请求、批量导出或同步调用。
- 是否影响页面首屏、弹窗打开、导出耗时、定时任务耗时。
- 性能风险只需输出验证建议和风险点，不要求生成自动化性能测试建议。

## 无业务变更处理规则

- 如果 diff 仅包含纯文档、注释、格式化、空白行、日志文案、无业务含义的重命名或排版调整，不强行生成测试点。
- 对无业务变更，应说明判断依据和影响结论。
- 可输出“无需新增业务测试点，建议做最小冒烟或无需回归”的结论。
- 如果无法判断是否存在业务影响，应列为待确认点，不生成确定性预期。
- 如果文档变更会影响用户操作、接口契约、需求规则或验收标准，应按需求/接口影响继续分析。

## 对话输出流程

当执行 Git diff、commit 分析时，必须在对话中输出分析结果，同时保存本地文件。

对话中必须输出：

- 当前分支
- commit 对比范围
- 需求与 Diff 对照结论
- diff 涉及文件清单
- 核心改动点
- 影响功能范围
- 疑似风险点
- P0 测试点
- 回归测试范围
- 本地分析报告路径
- 本地 XMind 用例路径

如果 diff 中存在规则不明确、业务含义不明确、字段口径不明确、页面入口不明确的情况：

- 必须先在对话中列出待确认点。
- 暂停生成最终 XMind 用例。
- 等用户确认或跳过后再继续。
- 用户确认或跳过后继续执行时，仍必须在对话中输出完整 diff 分析摘要，不能只输出本地文件路径。
- 续跑后的对话至少包含当前分支、commit 对比范围、diff 文件清单、核心改动点、待确认点处理结果、疑似风险点、P0 测试点、回归范围、本地分析报告路径、本地 XMind 用例路径。

Git diff 分析对话输出默认使用以下结构：

```markdown
## 本次分析范围

## Commit 对比范围

## 需求与 Diff 对照结论

## Diff 涉及文件清单

## 核心改动点

## 影响功能范围

## 待确认点

## 疑似风险点

## P0 测试点

## 回归范围

## 本地文件
```

如果存在待确认点，则输出到“待确认点”后暂停，不继续生成最终用例。

用户确认或跳过待确认点后继续执行时，也必须重新按上述模板输出最终摘要，并在“本地文件”中列出报告和 XMind 用例路径。

## 影响范围分析规则

重点分析：

- 页面入口和用户操作路径
- 接口入参、出参、返回结构
- SQL 查询条件、关联关系、聚合口径、排序分页
- 权限、组织、部门、角色、数据范围
- 标签、状态、枚举、字典、筛选条件
- 汇总与明细一致性
- 历史数据和空数据兼容
- 默认参数、空值、多值、异常值
- 前端展示、导出、下钻、联动查询

## 风险点识别规则

重点识别：

- SQL 语法风险
- 空指针或参数为空风险
- where 条件漏加或误加
- join 条件导致漏数、重复数
- group by 粒度错误
- count、sum、avg、distinct 口径错误
- 排序字段变更导致页面排序异常
- 返回结构变更导致前端解析异常
- 公共接口变更导致多模块回归
- 权限和数据范围绕过
- 旧参数不兼容
- merge commit 对比范围不明确
- 接口契约变更未同步调用方
- 配置默认值或灰度范围不明确
- 大数据量查询、导出或聚合性能退化

风险等级：

- P0/P1/P2 判定参考 `docs/codex/qa-priority-rules.md`。
- 风险等级统一使用 P0/P1/P2，不引入 P3。
- 轻微文案、日志、低概率体验问题一般不单独生成 XMind TC，可作为备注、建议或最小冒烟项记录。

## 疑似缺陷识别规则

当 diff 与需求存在以下情况时，应输出“疑似缺陷”而不仅是普通风险点：

- 需求明确要求，但 diff 完全未覆盖。
- diff 实现方向与需求明确相反。
- 接口入参、出参、SQL 条件、权限条件缺失会导致核心验收失败。
- 新增字段只在页面展示，未同步导出、弹窗、下钻或接口。
- 页面和导出口径不一致，且需求要求一致。
- 汇总和明细口径不一致，且影响核心数据。
- 新旧参数、历史入口或调用方兼容性被破坏。
- 公共接口、公共 SQL、公共组件变更后未覆盖调用方。
- 需求明确要求权限、组织、角色、数据范围控制，但 diff 未体现相关控制。
- 需求明确要求异常处理、空数据处理、兼容处理，但 diff 未体现相关处理。
- SQL where、join、group by、聚合、去重、日期口径与需求明确规则不一致。
- 删除、重命名或调整字段后未同步前端展示、导出、接口契约或下游调用方。

疑似缺陷必须包含：

- 缺陷现象
- 需求证据
- diff 证据
- 影响范围
- 建议验证方式
- 是否阻塞提测

疑似缺陷输出约束：

- 证据不足时不能直接断定为缺陷，只能列为风险点或待确认点。
- 不能凭代码现状反推需求事实。
- 疑似缺陷应优先进入 P0/P1 测试点和回归范围。
- 如果缺陷判断依赖未确认需求口径，应标记为“疑似缺陷，需确认”，不能写成确定缺陷。
- 疑似缺陷输出应服务测试验证，不要求 Codex 直接修改业务代码。
- 除非用户明确要求修复，否则只输出缺陷分析、影响范围和验证建议。

## 回归范围判断规则

从以下维度判断：

- 直接修改的页面、接口、SQL、报表
- 调用公共接口或公共 SQL 片段的上游页面
- 与变更字段、筛选条件、统计口径相关的同类模块
- 历史默认查询、带条件查询、分页、排序、导出
- 汇总页、明细页、弹窗、下钻页之间的数据一致性
- 不同角色、不同组织、不同部门的数据范围

回归范围必须分层输出：

- 核心回归：直接变更模块、核心接口、核心 SQL、核心权限、核心数据口径。
- 关联回归：调用同一公共逻辑、同一 SQL、同一接口、同一配置或同一字段口径的模块。
- 冒烟回归：历史入口、默认查询、基础跳转、基础导出、页面打开等低成本确认项。

不要把所有相关模块无差别列为同一优先级；必须说明每一层的回归原因。

## 本地输出规则

Git diff 分析需要拆成两个文件：

- 分析报告保存到：`testcases/diff/reports/`
- 可直接导入 XMind 的 Markdown 用例保存到：`testcases/diff/xmind/`

文件命名建议：

- 单 commit 分析报告：`diff_<commit_short>_analysis_<yyyyMMdd_HHmmss>.md`
- 单 commit XMind 用例：`diff_<commit_short>_xmind_<yyyyMMdd_HHmmss>.md`
- 双 commit 分析报告：`diff_<old_commit_short>_to_<new_commit_short>_analysis_<yyyyMMdd_HHmmss>.md`
- 双 commit XMind 用例：`diff_<old_commit_short>_to_<new_commit_short>_xmind_<yyyyMMdd_HHmmss>.md`

分析报告可以包含：

- 本次分析范围
- commit 对比范围
- diff 涉及文件清单
- 核心改动点
- 影响功能范围
- 疑似风险点
- P0 测试点
- 回归测试范围
- XMind 用例文件路径

XMind 用例文件必须严格遵守 `docs/codex/xmind-case-rules.md`，文件内容必须直接从 `- 根节点名称` 开始，不能包含分析报告标题、解释说明、总结说明或代码块。

如果选择“只分析”模式，只生成分析报告并在报告中说明未生成 XMind 用例。

如果选择“生成用例”模式，只生成 XMind 用例文件；如需要记录分析依据，可在 `testcases/index.md` 的备注中简述。

每次生成本地分析报告或 XMind 用例后，必须更新 `testcases/index.md`，记录生成时间、来源类型、commit 范围、报告路径、XMind 用例路径和备注。
