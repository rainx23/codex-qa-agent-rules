# XMind Markdown 测试用例规则

## 适用场景

当用户要求生成测试点、测试用例、XMind 用例、可导入 XMind 的 Markdown 时，按本规则执行。

## 默认边界

- 默认以资深测试专家、测试架构师身份设计用例。
- 默认只做测试分析和用例设计，不修改任何业务代码。
- 用例应基于需求、Git diff、OpenSpec、禅道需求或用户提供的上下文生成。
- 信息不足时，输出待确认点，不把推断写成确定预期。
- 证据不足时，不生成确定性预期，不虚构 SQL、字段、接口、页面入口。

## 输出模式和范围规则

- 支持三种输出模式：只分析、生成用例、完整输出。
- XMind 用例规则仅约束“生成用例”和“完整输出”中的 XMind Markdown 用例部分。
- 只分析模式下，不生成 XMind 用例正文。
- 支持两种用例范围：只列 P0、完整用例。
- 只列 P0：只生成 P0 核心风险、核心数据口径、核心链路相关 TC。
- 完整用例：覆盖 P0/P1/P2 关键场景，但仍必须去重，禁止低价值、无业务价值、重复语义用例。
- P0/P1/P2 判定统一参考 `docs/codex/qa-priority-rules.md`。
- 查询、列表、弹窗、跳转、导出、接口、SQL、权限等通用关注点可参考 `docs/codex/qa-common-templates.md`，但不得机械展开为低价值 TC。

## XMind 根节点强制规则

- 最终 XMind 用例必须包含一个总标题根节点。
- 根节点必须只有一个。
- 根节点名称需根据需求自动生成。
- 根节点名称必须具备业务语义。
- 根节点禁止使用“测试用例”这种无业务意义名称。
- 根节点禁止为空。
- 所有 TC 必须挂载在根节点下面的测试维度中。
- 禁止直接从 TC001 开始输出。

## 固定测试维度

一级节点必须按测试维度分组。测试维度名称固定，不能自由发挥。按需求类型自动选择以下维度，不相关的维度可以不输出：

- 功能测试
- 数据测试
- 排序测试
- 导出测试
- 异常测试
- 权限测试
- 兼容性测试
- 回归测试
- SQL验证

## 分组规则

- 不允许所有 TC 平铺输出。
- 每个测试维度必须单独作为一级节点。
- TC 必须挂载在对应测试维度下面。
- 不允许 TC 直接挂根节点。
- 同一测试维度下，如果多个 TC 拥有相同页面入口、模块入口、弹窗入口、列表入口或业务对象，必须优先抽取为公共入口节点，再把 TC 挂在该公共入口节点下，避免每条 TC 重复展示相同入口造成视觉疲劳。
- 公共入口节点必须使用业务化名称，例如“正式和模拟交易临底汇总卡T0股底仓弹窗”“首页与弹窗资金源组组合”，禁止使用“模块”“入口”“分组”这类空泛词。
- 公共入口节点只承载相同入口或相同业务对象，不得把验证点不同但入口无关的 TC 强行合并到一个节点。
- 相同维度场景必须归类到同一测试分组。
- 没有相关场景的维度可不输出。

## 严格 XMind 层级结构

默认必须严格使用以下纯业务层级：

- 根节点名称
    - 测试维度
        - 公共入口或业务对象
            - TC001
                - 二级功能点
                    - 操作步骤
                        - 预期结果

如果某个测试维度下不存在可复用的公共入口或业务对象，也允许省略公共入口层，直接使用以下结构：

- 根节点名称
    - 测试维度
        - TC001
            - 一级业务模块
                - 二级功能点
                    - 操作步骤
                        - 预期结果

示例结构只能作为结构参考，不要在实际用例中输出“一级业务模块”“二级功能点”“操作步骤”“预期结果”这些模板词，实际内容必须是业务化描述。

## 严格格式要求

- 必须使用 Markdown 的 `-` 层级结构。
- 所有缩进统一使用 4 个空格。
- 禁止使用 tab。
- 禁止使用 table。
- 禁止使用 JSON。
- 禁止输出代码解释。
- 禁止输出“以下是”。
- 禁止输出总结。
- 禁止输出说明文字。
- 禁止输出示例说明。
- 禁止多个内容拼接成一行。
- 禁止使用 Markdown 代码块。
- 最终输出必须适合直接导入 XMind。

## 严禁出现以下格式

禁止出现标签式节点：

- `模块：xxx`
- `测试点：xxx`
- `测试步骤：xxx`
- `预期结果：xxx`
- `前置条件：xxx`
- `操作步骤：xxx`
- `优先级：xxx`

禁止出现压缩格式：

- `TC001-模块-步骤-结果`

禁止出现模板化层级：

- `TC001`
    - `模块：xxx`
        - `测试点：xxx`
            - `步骤：xxx`
                - `预期：xxx`

必须使用纯业务层级：

- `TC001`
    - `功能模块业务描述`
        - `功能点业务描述`
            - `具体操作业务描述`
                - `具体结果业务描述`

## 测试编号规则

- 测试编号必须从 TC001 开始。
- 测试编号必须连续递增。
- 不允许重复编号。
- 不允许跳号。
- 所有测试维度共用同一套连续编号。
- 编号顺序必须正确。

## 测试设计原则

- 必须从高级测试专家角度设计测试用例。
- 一个 TC 只验证一个核心测试点。
- 自动补充遗漏场景。
- 自动去重。
- 用例节点必须短文本表达，优先使用“条件 + 动作”和“结果断言”的精简业务语句，避免长句、报告式描述和重复铺垫。
- 操作步骤只保留触发用例所需的关键条件和动作，不重复书写已在上层公共入口节点表达过的模块、页面、弹窗信息。
- 预期结果只保留核心断言，优先写参数是否携带、取值范围、状态变化、数据一致性等可验证结果，不写解释性背景。
- 预期结果必须可观察、可验证，禁止使用“正常”“正确”“合理”“符合预期”等无法直接判定通过/失败的模糊词。
- 需求未明确提示文案、错误文案、展示文案时，不得在预期中写死具体文案，应将文案口径列为待确认点；用例中仅验证可观察行为或状态。
- 生成用例前必须主动识别并删除重复、低价值或已被其他 TC 覆盖的用例，例如单纯验证按钮可点、页面可打开、模块规则一致但已在核心 TC 中覆盖的重复回归。
- 删除重复用例后，必须检查是否遗漏高风险场景；优先补充数据边界、条件组合、异常输入、历史兼容、二次入口、状态未生效、聚合口径差异等容易线上出问题的场景。
- 优先从业务风险角度设计测试用例，而不是仅从页面功能角度设计。
- 优先覆盖最容易出现线上问题的场景。
- 优先覆盖最容易出现数据错误的场景。
- 优先覆盖最容易遗漏的场景。
- 优先覆盖最容易引发业务事故的场景。
- 仅覆盖与需求或 diff 强相关的业务测试点。
- 禁止补充无意义、低价值、明显重复、脱离业务的泛化场景。
- 禁止生成显而易见的低质量用例，例如点击按钮验证按钮可点击、页面正常打开、页面正常展示。
- diff 中纯文档、注释、格式化、日志等无业务变更时，不强行生成 XMind 测试用例。
- 证据不足时，只能围绕风险点和待确认点输出，不生成确定性业务预期。

## 用例去重和删减规则

生成 XMind 前必须先做去重和删减：

- 相同入口、相同条件、相同动作、相同预期的 TC 必须合并。
- 仅模块名称不同但规则一致的 TC 必须合并，例如正式交易和模拟交易规则一致时合并为一条。
- 同一字段组具备相同校验规则时，可合并为一个字段组验证 TC。
- 已被核心参数组合覆盖的单条件重复用例应删除。
- “页面打开”“按钮可点击”“列表展示正常”“查询成功”这类无明确业务断言的用例应删除。
- 删除重复用例后必须检查是否遗漏高风险场景，例如空集合、未点击查询、二次入口、历史参数、权限继承、接口兼容。

## 最小有效用例集规则

- 生成完整用例前，必须先按风险点构建最小覆盖集合。
- 多个条件组合时，优先覆盖能区分不同结果的最少组合，不穷举所有排列。
- 同一接口、同一 SQL、同一字段组、同一入口下的相同验证逻辑必须合并。
- 已被 P0/P1 主链路覆盖的低价值 P2 场景，不再单独生成。
- 每条 TC 必须能回答：验证什么风险、为什么需要单独存在、失败后能定位什么问题。
- 如果某条 TC 不能对应明确需求点、diff 证据、业务风险或历史缺陷，应删除或合并。
- 不为了凑数量生成 P1/P2。
- 不为了覆盖模板清单机械生成低价值用例。
- 保留能暴露缺陷、能验证核心规则、能定位风险来源的用例。

## 用例变更管理规则

- 生成新用例前，应查看 `testcases/index.md` 是否已有同一需求、同一标题或同一 commit 范围的历史产物。
- 如果是同一需求再次生成，应优先判断本次是更新、补充、替代还是废弃旧版本。
- 新版本用例文件不覆盖旧文件，使用新的时间戳生成。
- `testcases/index.md` 备注中应说明版本关系，例如“替代上一版”“补充 P0”“根据用户确认更新”“转换失败仅保留 md”。
- 若旧用例存在重复或无效场景，新版应在分析报告中说明优化点，而不是生成一份视觉上完全相同的文件。

## 测试方法选用规则

生成用例前应按功能点扫描以下触发条件，命中时优先使用对应测试设计方法；不为凑方法数量生成低价值用例。

- 等价类划分：存在格式、类型、范围限制的输入字段。
- 边界值分析：存在明确数值、长度、数量、比例、金额、时间上下限。
- 决策表：存在两个及以上条件组合，且不同组合产生不同结果。
- 状态迁移：存在明确状态字段、状态流转、审批流、工单流或终态。
- 错误猜测：历史容易出问题、逻辑分支超过 3 层、空值或异常链路风险高。
- 风险驱动：涉及金额、权限、核心流程、状态变更、外部接口或数据一致性。
- 用户路径分析：跨越两个及以上模块的端到端流程。
- 历史缺陷回归：本次变更范围与历史缺陷模块或修复范围重叠。

## 等价类和边界值增强规则

以下字段类型命中时，应优先考虑等价类、边界值或条件组合：

- 输入框：空值、最小长度、最大长度、超长、特殊字符、前后空格、大小写、模糊匹配。
- 数值：0、负数、最小值、最大值、超范围、小数精度、四舍五入、金额单位。
- 日期：开始日期、结束日期、同一天、跨月、跨年、未来日期、历史日期、交易日和非交易日。
- 枚举：全部枚举、无效枚举、默认枚举、历史枚举、字典映射缺失。
- 多选：未选择、单选、多选、全选、取消、交集为空、交集非空、选择后未查询。
- 查询条件组合：外层条件与弹窗条件、列表条件与跳转参数、导出条件与页面条件。

仅当这些边界与当前需求或 diff 强相关时才生成 TC。

## 反向用例规则

反向用例必须来自真实业务风险，优先覆盖：

- 必填参数缺失。
- 参数类型错误。
- 参数值越界。
- 多选交集为空。
- 已选择条件但未点击查询。
- 无权限访问、越权参数、跨组织数据。
- 接口失败、上游数据缺失、返回空数组或 null。
- 重复提交、重复请求、状态非法。
- 历史旧参数不兼容。

禁止为无业务意义的随机非法字符、随机超长输入生成大量低价值异常 TC。

## 风险优先级判断参考

设计 P0/P1/P2 用例时，优先参考以下风险维度，不强制输出评分：

- 业务影响：是否影响收入、订单、交易、核心流程。
- 用户影响：影响用户范围和使用频率。
- 逻辑复杂度：分支、条件组合、状态流转是否复杂。
- 历史缺陷：相关模块是否曾频繁出问题。
- 外部依赖：是否依赖第三方、上下游系统、接口、定时任务。
- 数据敏感性：是否涉及金额、库存、权限、客户数据、策略数据。
- 权限风险：是否存在越权、数据隔离、角色差异风险。
- 变更范围：是否涉及公共接口、公共 SQL、公共组件或多模块复用逻辑。

## 需求解析规则

生成测试用例前，先理解：

- 需求改动点
- diff 变更点
- 需求与 diff 的对应关系
- 影响范围
- 核心业务逻辑
- 数据来源
- 展示逻辑
- 计算逻辑
- 导出逻辑

然后自动分析需求属于：

- 列表类
- 表格类
- 导出类
- 配置类
- 流程类
- 权限类
- 报表类
- 数据同步类
- 统计类
- 查询类
- 弹窗类
- 下钻类

再生成对应测试场景。

## 需求与 Diff 联合用例规则

- 如果同时存在需求内容和 diff 内容，XMind 用例必须基于“需求规则 + diff 实现 + 二者差异”联合生成。
- 不允许只根据代码 diff 生成用例而忽略前置需求规则。
- 不允许只根据需求生成用例而忽略 diff 已经暴露的实现风险。
- 每个核心需求点应尽量映射到对应的 diff 实现、测试场景或待确认点。
- 如果需求与 diff 不一致，必须优先生成风险验证类用例或待确认点，不得把代码实现写成确定性需求预期。
- 如果需求未说明但 diff 新增了字段、接口、SQL、页面逻辑，应作为“需求外变更风险”处理，优先列入回归或待确认。

## 字段类需求专项规则

如果需求涉及字段新增、字段修改、字段展示、字段变更，必须先扫描以下风险维度；只有与当前需求、diff、代码证据或业务风险强相关时才生成 TC，未命中真实风险的维度不单独生成用例：

- 字段展示
- 字段名称
- 字段顺序
- 默认状态
- 字段来源
- 字段取值
- 字段映射
- 枚举值、状态值、字典值映射
- code/name 映射
- 空值
- 0 值
- 负数
- 特殊值
- 超长值
- 特殊字符
- 多值展示
- +N 展示
- 精度
- 单位
- 格式
- 四舍五入
- 排序
- 导出
- 列表、详情、导出、汇总中的口径一致性

相同字段、相同入口、相同规则、相同预期的用例必须合并；相同测试逻辑的多个字段，允许合并为一个高质量测试场景，避免机械重复。

## 列表/表格类需求专项规则

如果需求涉及列表、表格、汇总卡、明细页、数据展示页，必须先扫描以下风险维度；只有与当前需求、diff、代码证据或业务风险强相关时才生成 TC，未命中真实风险的维度不单独生成用例：

- 默认查询结果
- 查询条件筛选
- 多条件组合筛选
- 分页切换
- 每页条数切换
- 列展示
- 列顺序
- 列宽
- 固定列
- 空数据展示
- 大数据量展示
- 汇总行、合计行、明细行一致性
- 点击行
- 链接
- 下钻
- 弹窗
- 页面刷新
- 分页后排序一致性

## 数据场景覆盖规则

数据类、金融类、交易类、报表类需求必须先扫描以下风险维度；只有命中真实需求、diff、SQL、代码证据或业务风险时才生成对应测试点：

- 数据来源
- 字段取值
- 字段映射
- code/name 映射
- 数据解析
- 数据转换
- 日期逻辑
- 状态逻辑
- 最大值逻辑
- 最小值逻辑
- 聚合逻辑
- 重复数据
- 金额
- 数量
- 比例
- 精度
- 聚合
- 去重
- 排序
- 空值
- 多值
- 异常状态
- 时间范围边界
- 汇总与明细一致性
- 历史数据兼容性
- 数据权限和组织范围

## 排序场景覆盖规则

排序相关需求必须先扫描以下风险维度；只有需求、diff 或代码证据涉及排序口径、排序字段、排序联动或历史排序兼容时才生成 TC：

- 默认排序
- 升序排序
- 降序排序
- 升降序切换
- 排序稳定性
- 空值排序
- 相同值二级排序
- 数字、金额、日期、文本字段排序
- 翻页后排序是否保持
- 导出结果排序是否与页面一致

## 导出场景覆盖规则

只有需求涉及导出、diff 改动影响导出、或页面/接口/SQL 口径可能影响导出时才生成导出 TC。命中导出风险时先扫描以下维度：

- 默认条件导出
- 筛选条件后导出
- 排序后导出
- 空数据导出
- 大数据量导出
- 导出字段名称
- 导出字段顺序
- 导出字段值
- 页面数据与导出数据一致
- 页面排序与导出排序一致
- 页面展示简写与导出完整值一致
- 金额、数量、比例、日期格式正确
- +N 导出逻辑
- 无权限用户不可导出

## 异常场景覆盖规则

异常场景必须来自真实业务风险，禁止随机生成大量非法输入或脱离业务的异常测试。只有与当前需求、diff、接口契约、SQL 口径、权限或历史缺陷强相关时才生成异常 TC；命中异常风险时先扫描以下维度：

- 空值
- null
- 必填参数为空
- 非法参数
- 参数类型错误
- 特殊字符
- 超长文本
- 日期范围非法
- 查询无数据
- 接口超时或失败
- 上游数据缺失
- 权限不足
- 重复数据
- 重复提交或重复请求
- 异常状态
- 边界值

异常场景生成前应先判断是否已有功能、数据或接口 TC 覆盖相同风险；如已覆盖，不再重复生成。

## 权限场景覆盖规则

只有需求、diff 或业务风险涉及用户、角色、组织、部门、数据范围、导出权限、越权或数据隔离时才生成权限 TC。命中权限风险时先扫描以下维度：

- 有权限用户可正常查看或操作
- 无权限用户不可查看或操作
- 不同组织、部门、角色的数据隔离
- 越权访问拦截
- 导出权限控制
- 下钻或明细页权限继承

## 兼容性和回归场景覆盖规则

必须根据需求或 diff 影响先扫描以下兼容和回归风险；未命中真实影响的维度不单独生成 TC：

- 历史数据兼容
- 旧参数兼容
- 默认查询兼容
- 历史入口兼容
- 原有列表、详情、导出、排序、筛选功能回归
- 公共接口或公共 SQL 影响的关联模块回归

## SQL验证用例规则

如果需求、diff 或代码证据涉及真实数据库字段、数据查询、数据展示、数据一致性、排序、导出、日期逻辑、状态逻辑、聚合逻辑，必须扫描 SQL 验证风险；只有存在真实表、字段、SQL 口径或可验证数据链路证据时才生成 SQL验证 类用例。

SQL 验证用例必须具备真实验证意义。SQL 字段、条件、聚合逻辑必须来源于需求、diff 或已知代码，不允许虚构字段。信息不足时，只能输出 SQL 验证思路，不能编造具体 SQL。

SQL 要求：

- sql 必须作为 XMind 节点内容输出。
- sql 关键字全部小写。
- 禁止 tab。
- 统一使用空格缩进。
- 字段逗号前置。
- where 条件 and 对齐。
- sql 必须适合 Markdown / XMind。
- SQL 可以写成普通 XMind 节点内容，禁止使用 Markdown 代码块。

SQL 格式结构示例：

- `SQL验证`
    - `TC008`
        - `数据口径验证`
            - `使用需求或代码中的真实表字段编写 SQL 校验数据来源、筛选条件、聚合口径`
                - `查询结果与页面展示、导出结果、需求规则保持一致`

## 禁止脑补规则

禁止假设需求文档、diff 或代码中未提及的：

- 页面功能
- 权限逻辑
- 接口逻辑
- 缓存逻辑
- 审批流程
- 配置逻辑
- 系统能力

如测试判断依赖这些信息，必须列为待确认点，不允许写成确定预期。

## 待确认点与用例生成关系

- 如果存在未确认需求点，且用户尚未确认或跳过，不允许生成最终 XMind Markdown 用例。
- 如果用户已确认，则按确认后的规则生成用例。
- 如果用户明确跳过，则允许生成用例，但不能把未确认内容写成确定性预期。
- 对未确认但需要覆盖的风险，只能以“验证页面展示与最终确认规则一致”“验证导出结果与确认口径一致”这类方式表达。
- 禁止虚构字段、SQL、页面入口、权限逻辑、导出口径。
- SQL 验证用例中，如果表名、字段名、关联关系不明确，只能写 SQL 验证思路，不能编造具体 SQL。

## 对话展示与本地文件格式区分

- 对话中可以展示测试点摘要、P0 测试点、回归范围、待确认点。
- 可直接导入 XMind 的完整 Markdown 用例必须保存到本地文件。
- 如果用户明确要求“在对话中展示完整 XMind Markdown / 输出 md / 给我代码块格式 / 把 XMind Markdown 也打印出来”，对话中必须使用 markdown 代码块包裹，方便复制。
- 对话中的代码块只是展示方式，不能写入本地 XMind Markdown 文件。
- 默认情况下，对话中不强制打印完整 XMind Markdown，避免内容过长。
- 如果用户只是要求保存本地，不要求打印完整 Markdown，对话中只输出摘要和文件路径即可。
- 用户确认或跳过待确认点后继续生成 XMind 文件时，对话中必须展示测试点摘要、P0 重点、回归范围和本地文件路径，不能只输出已生成的文件清单。
- 本地 XMind Markdown 文件必须直接从 `- 根节点名称` 开始。
- 本地 XMind Markdown 文件用于转换 `.xmind`，必须保持纯 `-` 层级结构，禁止包含分析报告标题、解释说明、总结说明、表格、JSON、代码块。
- 有待确认点且用户未确认或未跳过时，不允许生成最终 XMind Markdown 文件。
- 对话展示和本地文件格式规则必须区分处理，不能互相冲突。
- 对话中如需展示片段，应使用普通摘要或少量片段；本地文件必须保持纯 XMind Markdown，不包含代码块包裹。

## 本地 XMind 文件输出规则

- 可直接导入 XMind 的 `.md` 文件必须直接从 `- 根节点名称` 开始。
- 文件中禁止出现分析报告标题。
- 文件中禁止出现解释说明。
- 文件中禁止出现总结说明。
- 文件中禁止使用 Markdown 代码块。
- diff 生成的 XMind Markdown 用例保存到 `testcases/diff/xmind/`。
- 禅道需求生成的 XMind Markdown 用例保存到 `testcases/zentao/xmind/`。
- 每次生成本地 XMind Markdown 用例后，必须使用 `scripts/md_to_xmind.py` 转换生成可区分命名的 `.xmind` 文件，`.md` 和 `.xmind` 保存到同一目录。
- 转换命令：`python scripts/md_to_xmind.py <XMind Markdown 用例文件路径>`。
- 转换完成后必须检查 `.xmind` 文件已生成，并至少验证包内包含 `content.json`、`metadata.json`、`manifest.json`。
- 每次生成本地 XMind 用例文件后，必须更新 `testcases/index.md`，记录 Markdown 用例路径和 `.xmind` 文件路径及其对应的分析报告路径或来源。

## 输出前自检规则

输出或保存前必须自检：

- 是否覆盖所有 P0 风险。
- 是否存在重复 TC。
- 是否存在无明确业务断言的 TC。
- 是否存在证据不足却写死预期的 TC。
- 是否存在只验证页面打开、按钮可点、列表正常展示的低价值 TC。
- 是否所有待确认点都已进入分析报告。
- 是否只有一个业务根节点。
- 根节点是否有明确业务语义，且不叫“测试用例”。
- 一级节点是否都是固定测试维度。
- TC 是否全部挂在测试维度下面。
- TC 编号是否从 TC001 连续递增。
- 是否存在跳号、重复编号。
- 是否只使用 `-` 层级结构。
- 缩进是否统一为 4 个空格。
- 是否使用了 tab。
- 是否包含表格、JSON、代码块、解释说明或总结说明。
- 是否存在“模块：xxx”“测试点：xxx”“测试步骤：xxx”“预期结果：xxx”“前置条件：xxx”等标签式节点。
- 是否每个 TC 只验证一个核心测试点。
- 同一测试维度下，是否已把重复出现的相同入口、页面、弹窗、列表或业务对象抽成公共节点。
- 操作步骤和预期是否足够短，是否删除了不影响理解的铺垫文字。
- 是否存在低质量、重复语义、无业务价值用例。
- 是否删除了已被其他 TC 覆盖的重复用例，并补充了遗漏的高风险场景。
- 是否把待确认需求点写成确定预期。
- 是否适合直接导入 XMind。
- 是否存在仅模块名称不同、测试规则和预期完全一致的重复 TC；如存在，必须合并。
- 是否已将 XMind Markdown 转换为可区分命名的 `.xmind` 文件并完成基础结构检查。
- 是否 XMind 文件满足唯一根节点、固定维度、TC 连续编号、4 空格缩进。
- 是否已经删除仅模块名称不同但验证逻辑完全相同的重复用例。
- 是否存在由“必须覆盖”导致的机械穷举用例，如存在必须删除或合并。
- 是否存在未命中真实风险却生成的异常用例。
- 是否存在为了模板完整性强行生成的低价值导出、权限、兼容性用例。
- 是否所有 SQL 验证都基于真实表字段或明确说明为验证思路。
- 是否所有预期结果都可观察、可验证。
- 是否所有未确认口径都没有写成确定性预期。

## 同规则多模块用例合并规则

- 当多个模块、端、页面或业务类型的需求规则、操作路径、输入条件和预期结果完全一致，仅模块名称不同，例如“正式交易”和“模拟交易”规则一致时，必须优先合并为一条 TC，不允许为每个模块机械拆分重复用例。
- 合并后的 TC 应在业务模块节点或操作步骤中明确写出覆盖范围，例如“正式和模拟交易临底”或“分别在正式交易和模拟交易中执行”，预期结果中说明两个模块均需满足同一规则。
- 只有当不同模块存在数据源、字段口径、权限、页面入口、异常处理、参数规则、导出口径或历史兼容差异时，才允许拆分为独立 TC。
- 输出前自检时必须检查是否存在仅模块名称不同、验证点完全相同的重复 TC；如存在，应合并并保持 TC 编号连续。

## Markdown 转 XMind 文件规则

- 生成用例或完整输出模式下，最终产物必须同时包含可导入 XMind 的 Markdown 文件和由该 Markdown 转换出的 `.xmind` 文件。
- 默认使用仓库脚本 `scripts/md_to_xmind.py` 进行转换，不新增第三方依赖。
- `.xmind` 文件命名必须与 Markdown 用例文件明显区分：当 Markdown 文件名包含 `_xmind_` 时，转换后的 `.xmind` 文件名使用 `_workbook_` 替换第一个 `_xmind_`；其他情况在原文件名后追加 `_workbook`。
- 如果转换失败，必须在对话和分析报告或索引备注中说明失败原因，不能只返回 Markdown 用例。
- 如果用户明确要求只生成 Markdown 或暂不转换，可以跳过 `.xmind`，但必须在输出中说明这是用户指定的例外。

## XMind 转换失败处理规则

执行 `scripts/md_to_xmind.py` 失败时，必须：

- 保留已生成的 Markdown 用例文件。
- 输出失败命令、失败原因和是否已生成 `.xmind`。
- 检查常见原因：缩进不是 4 个空格、使用 tab、根节点不唯一、存在非 `-` 列表、文件编码异常、路径不存在。
- 修复 Markdown 格式后重新转换。
- 如果仍失败，在 `testcases/index.md` 备注中标记“xmind 转换失败，仅保留 md”，并在对话中明确说明。
- 不能在 `.xmind` 未生成时宣称已完成 XMind Workbook。
