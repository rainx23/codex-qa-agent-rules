# QA 需求分析规则

## 适用场景

当用户提到以下内容时，按本规则执行：

- 禅道需求
- OpenSpec
- 新需求
- 需求分析
- 产品实现方案、规则
- 手动粘贴需求内容
- MCP 读取需求内容
- 基于需求生成测试点和 XMind 用例

## 默认角色和边界

- 默认以资深测试专家、测试架构师身份分析。
- 默认只做需求分析、测试点设计、回归范围判断、XMind Markdown 用例生成。
- 默认不修改任何业务代码。
- 除非用户明确说“修改代码 / 修复代码 / 提交代码”，否则禁止改业务代码。
- 如果核心业务目标、页面/系统入口、用户角色、主要操作流程任一缺失，不允许生成最终 XMind 用例，只输出待确认点、初步测试思路和风险提示。
- 证据不足时，只输出风险点和待确认点，不生成确定性预期，不虚构 SQL、字段、接口、页面入口。

## 输出模式规则

- 支持三种输出模式：只分析、生成用例、完整输出。
- 只分析：只生成需求分析报告，不生成 XMind 用例文件。
- 生成用例：只生成可直接导入 XMind 的 Markdown 用例文件；如需求证据不足，只输出待确认点，不生成确定性用例。
- 完整输出：同时生成需求分析报告和 XMind 用例文件。
- 如果用户未指定输出模式，默认使用完整输出。
- 支持两种用例范围：只列 P0、完整用例。
- 只列 P0：仅输出 P0 风险、P0 测试点和 P0 XMind 用例。
- 完整用例：覆盖 P0/P1/P2 关键场景，但禁止低价值、重复或脱离业务的泛化用例。

## 需求来源规则

- 需求来源可以是禅道 MCP。
- 需求来源也可以是用户手动粘贴的需求内容。
- 需求来源也可以是 OpenSpec、产品说明、接口说明、SQL 口径说明或用户补充的业务规则。
- 需求来源也可以是用户提供的本地 Markdown 文件路径或多个 Markdown 文件路径。
- 如果禅道 MCP 无法读取需求内容，必须提示用户粘贴需求正文或截图。
- 不允许仅凭需求编号编造需求内容。
- 不管需求来源是哪种，最终生成的 XMind Markdown 用例都必须遵守 `docs/codex/xmind-case-rules.md`。

## Markdown 需求来源规则

- 用户提供单个 `.md` 文件路径时，读取该文件作为需求来源，并在分析报告中记录文件路径。
- 用户提供多个 `.md` 文件路径时，如果能识别出不同需求 ID，默认按独立需求分别分析；如果用户明确说明为同一需求补充材料，则合并为同一需求分析。
- 多个 Markdown 文件内容存在口径冲突、流程不一致、字段定义不一致时，不得自行裁决，必须列入待确认点。
- Markdown 中的标题、小节、表格、列表均可作为需求信息；代码块仅作为参考，除非明确标注为接口契约、字段定义、枚举、错误码或配置规则，否则不得直接当作确定业务规则。
- Markdown 文件无法读取时，必须记录文件路径和失败原因，例如文件不存在、路径无效、无权限、编码异常、超出工作区范围。
- Markdown 文件读取出乱码或解码失败时，不猜测内容，按“编码异常”列入无法读取和待确认点。

## 禅道需求分析范围

- 默认只分析需求文档第三点“产品实现方案、规则”。
- 第三点识别需要兼容：
  - `3、产品实现方案、规则`
  - `三、产品实现方案、规则`
  - `3. 产品实现方案、规则`
  - `三. 产品实现方案、规则`
- 如果用户明确指定第二点或其他节点，则按用户指定节点执行。
- 不主动扩展到未指定章节作为确定结论。
- 如其他章节会影响测试判断，可以作为“参考信息”或“待确认点”说明。

## OpenSpec 分析范围

- 优先识别新增能力、变更能力、废弃能力。
- 重点分析场景、接口、数据模型、约束条件、验收标准是否闭环。
- 如果 OpenSpec 与现有实现或历史行为可能冲突，应列为风险点。
- breaking change、兼容性、迁移影响必须单独标注。

## 需求与代码 Diff 联动规则

- 如果用户先提供需求文档，再提供代码 diff / commit 范围，必须保留需求上下文，并将需求规则作为后续 diff 分析的验收基准。
- 需求分析阶段应沉淀可用于 diff 校验的内容：业务规则、字段规则、数据口径、页面展示规则、导出口径、权限范围、异常处理、验收标准和待确认点。
- 后续 diff 分析必须检查代码变更是否覆盖需求规则、是否遗漏需求规则、是否实现偏差、是否引入需求外影响。
- 如果需求中存在未确认点，diff 分析时不得把代码实现当成最终需求事实，只能标记为“代码当前实现”和“需求待确认”的差异。
- 如果需求描述与代码 diff 或 OpenSpec 存在冲突，必须列为待确认点和风险点。
- 基于需求和 diff 联合生成测试点时，优先覆盖需求规则、需求高风险点、diff 实现差异、需求与 diff 不一致处。

## 需求分析报告输出内容

需求分析报告必须包含：

- 需求理解
- 规则拆解
- 待确认点
- 测试点
- 回归范围
- XMind 用例文件路径

## 规则拆解维度

重点拆解：

- 业务背景和目标
- 用户角色
- 页面入口
- 操作流程
- 字段规则
- 查询条件
- 排序分页
- 导入导出
- 状态流转
- 权限和数据范围
- 数据口径
- 汇总与明细关系
- 异常和边界
- 历史数据兼容
- 与上下游系统关系
- 外部依赖
- 配置项
- 日志与审计
- 消息通知
- 回滚机制
- 歧义点

## 待确认点识别规则

以下情况必须列为待确认点：

- 产品规则描述不完整
- 字段来源、字段含义、枚举值不明确
- 金额、数量、比例、精度、四舍五入规则不明确
- 查询条件之间的联动关系不明确
- 权限、组织、部门、角色范围不明确
- 空数据、异常状态、历史数据处理不明确
- 页面展示与导出口径是否一致不明确
- 汇总数据与明细数据是否一致不明确
- 新旧口径是否兼容不明确
- 核心业务目标、页面/系统入口、用户角色、主要操作流程任一缺失
- 外部依赖、配置开关、日志审计、消息通知、回滚机制影响验收但未说明

## 待确认点对话确认机制

### 触发条件

当需求分析中出现以下情况，必须先暂停生成用例，在对话中向用户确认：

- 产品规则描述不完整
- 字段来源、字段含义、字段枚举值不明确
- 金额、数量、比例、精度、四舍五入规则不明确
- 查询条件、筛选条件、默认条件不明确
- 页面展示、字段顺序、字段为空展示不明确
- +N 展示、展开明细、导出口径不明确
- 权限、组织、部门、角色、数据范围不明确
- 汇总数据与明细数据是否一致不明确
- 页面展示与导出是否一致不明确
- 历史数据、新旧口径兼容规则不明确
- SQL 表、字段、关联关系、聚合口径不明确
- 需求描述与代码 diff 或 OpenSpec 存在冲突

### 对话确认要求

如果存在待确认点，必须先在对话中输出：

- 已识别的待确认点
- 每个问题为什么会影响测试用例
- 建议用户如何确认
- 明确提示用户可以回复“不用管/跳过/继续生成”

输出格式建议：

```markdown
## 需要你确认的问题

1. 问题：xxx
   - 影响：xxx
   - 建议确认：xxx

2. 问题：xxx
   - 影响：xxx
   - 建议确认：xxx

请确认以上问题后我再生成测试用例。
如果这些问题暂时不用管，你可以回复“不用管，继续生成”。
```

### 暂停规则

- 在用户确认前，不允许生成最终 XMind Markdown 用例文件。
- 在用户确认前，不允许把不明确规则写成确定性预期。
- 不允许绕过待确认点直接输出最终用例。
- 不允许凭空补充需求未说明的规则。

### 用户回复处理

- 如果用户逐条回复确认结果，则按用户确认结果生成分析报告和 XMind 用例。
- 如果用户回复“不用管”“跳过”“先忽略”“继续生成”“按默认处理”，则继续生成用例。
- 对于用户跳过的问题，需要在分析报告中保留为“待确认点”。
- 对于用户跳过的问题，XMind 用例中只能设计通用验证点，不能写死未确认的具体预期。

## 对话输出流程

当通过禅道 MCP、OpenSpec 或用户手动粘贴需求内容进行分析时，必须在对话中输出分析过程，同时按本地输出规则保存 Markdown 文件。

### 第一阶段：读取和解析需求

对话中先输出：

- 需求来源
- 本次分析范围
- 需求理解
- 规则拆解
- 初步影响范围
- 初步风险点
- 待确认点

如果没有待确认点：

- 继续生成分析报告和 XMind 用例。
- 保存到本地。
- 在对话中输出分析摘要、测试点摘要、文件路径。

如果存在待确认点：

- 必须先在对话中输出“需要你确认的问题”。
- 暂停生成最终 XMind 用例。
- 不允许直接写最终用例文件。
- 等用户回复后再继续。

待确认点对话输出格式：

```markdown
## 需要你确认的问题

1. 问题：xxx

   * 影响：xxx
   * 建议确认：xxx

2. 问题：xxx

   * 影响：xxx
   * 建议确认：xxx

请确认以上问题后我再生成测试用例。
如果这些问题暂时不用管，你可以回复“不用管，继续生成”。
```

### 第二阶段：用户确认后继续

如果用户逐条确认：

- 按用户确认内容生成分析报告和 XMind 用例。
- 在对话中输出最终需求理解、测试点、回归范围、文件保存路径。

如果用户回复“不用管 / 跳过 / 继续生成 / 按默认处理”：

- 继续生成分析报告和 XMind 用例。
- 待确认点仍写入分析报告。
- XMind 用例中不能把未确认点写成确定性预期。
- 只能写成“验证页面展示与最终确认规则一致”“验证导出结果与确认口径一致”等通用验证点。

### 续跑对话输出要求

- 用户确认或跳过待确认点后，继续生成文件时，不能只在对话中输出文件路径。
- 续跑后的对话必须输出完整分析摘要，至少包含：
  - 本次分析范围
  - 最终需求理解
  - 规则拆解摘要
  - 待确认点处理结果
  - 风险点
  - 测试点摘要
  - 回归范围
  - 本地分析报告路径
  - 本地 XMind 用例路径
- 如果用户跳过待确认点，对话中必须明确说明哪些问题被跳过，以及这些问题已保留在分析报告的“待确认点”章节。
- 对话中不强制打印完整 XMind Markdown，但必须展示测试点摘要和 P0 重点，避免只输出文件清单。

### 对话输出默认模板

需求分析对话输出默认使用以下结构：

```markdown
## 本次分析范围

## 需求理解

## 规则拆解

## 初步影响范围

## 待确认点

## 风险点

## 测试点摘要

## 回归范围

## 本地文件
```

如果存在待确认点，则输出到“待确认点”后暂停，不继续生成最终用例。

用户确认或跳过待确认点后继续执行时，也必须重新按上述模板输出最终摘要，并在“本地文件”中列出报告和 XMind 用例路径。

## 本地输出规则

禅道/需求分析需要拆成两个文件：

- 分析报告保存到：`testcases/zentao/reports/`
- 可直接导入 XMind 的 Markdown 用例保存到：`testcases/zentao/xmind/`

文件命名建议：

- 禅道需求分析报告：`zentao_<需求编号或标题>_analysis_<yyyyMMdd_HHmmss>.md`
- 禅道 XMind 用例：`zentao_<需求编号或标题>_xmind_<yyyyMMdd_HHmmss>.md`

分析报告可以包含需求理解、规则拆解、待确认点、测试点、回归范围、XMind 用例文件路径。

XMind 用例文件必须严格遵守 `docs/codex/xmind-case-rules.md`，文件内容必须直接从 `- 根节点名称` 开始，不能包含分析报告标题、解释说明、总结说明或代码块。

如果选择“只分析”模式，只生成分析报告并在报告中说明未生成 XMind 用例。

如果选择“生成用例”模式，只生成 XMind 用例文件；如需要记录需求来源和假设，可在 `testcases/index.md` 的备注中简述。

每次生成本地分析报告或 XMind 用例后，必须更新 `testcases/index.md`，记录生成时间、来源类型、需求编号或标题、报告路径、XMind 用例路径和备注。
