# QA 优先级规则

## 适用范围

本规则用于需求分析、Git diff 分析、测试点输出和 XMind 用例生成中的 P0/P1/P2 判定。

当 `AGENTS.md`、`qa-requirement-analysis-rules.md`、`code-diff-review-rules.md` 或 `xmind-case-rules.md` 需要判断风险等级、用例优先级或回归优先级时，统一引用本文件。

## 基本原则

- 优先级用于表达测试风险和验证顺序，不要求在每条 XMind TC 中输出固定字段。
- 优先级必须基于需求证据、diff 证据、历史风险或明确业务影响判断。
- 证据不足时，不得强行判定为确定缺陷；应标记为待确认点或风险点。
- 同一风险如果同时满足多个级别，按更高优先级处理。
- 不为凑数量生成低价值 P1/P2 用例。

## P0 判定

满足任一条件时，优先按 P0 处理：

- 核心业务流程不可用，例如核心页面无法进入、核心查询失败、关键跳转失败。
- 核心数据错误，例如金额、数量、交易、库存、策略、统计结果或汇总明细口径严重偏差。
- 严重权限问题，例如越权查看、越权操作、数据隔离失效。
- 接口契约变更导致调用方报错、页面无法解析或主链路中断。
- SQL 条件、关联、聚合、去重、日期口径错误可能导致大范围错数。
- 新旧参数或历史入口不兼容，导致线上既有链路失败。
- 需求与 diff 明确冲突，且影响核心验收结果。
- 生产事故高发模块、历史缺陷核心模块再次变更，且影响范围无法收敛。

## P1 判定

满足任一条件时，优先按 P1 处理：

- 主要功能可用但关键规则、口径、筛选、排序、导出或状态展示存在偏差风险。
- 影响多个页面、接口、报表、弹窗、下钻或上游调用方，但不直接导致核心链路不可用。
- 多条件组合、状态流转、枚举映射、默认值、空值或多值逻辑复杂，容易遗漏。
- 需求规则明确，但 diff 只覆盖部分实现，存在疑似遗漏。
- 字段新增或返回结构变更可能影响前端展示、导出或兼容性。
- 权限、组织、部门、角色或数据范围存在边界风险，但未发现确定越权。
- 配置项、开关、外部依赖、定时任务或上下游数据变化影响主要场景。

## P2 判定

满足任一条件时，优先按 P2 处理：

- 局部展示、低频筛选、边界输入、空数据、异常提示、兼容回归等非主链路场景。
- 页面文案、字段顺序、列宽、+N 展示、弱交互体验存在可验证风险。
- 需求影响范围明确较小，且已有 P0/P1 用例覆盖主规则。
- 低频角色、低频组织、少量历史数据或特殊数据形态的补充回归。

## 不建议生成用例的场景

以下场景不应单独生成 TC，除非它们与本次需求或 diff 的真实风险直接相关：

- 单纯验证按钮可点击。
- 单纯验证页面可以打开。
- 单纯验证列表有数据展示。
- 与需求无关的泛化异常输入。
- 已被其他 TC 覆盖且没有新增风险的重复回归。
- 纯文档、注释、格式化、日志文案、无业务含义重命名。

## 回归优先级

- 核心链路、核心数据口径、权限隔离、接口契约和公共 SQL 影响范围优先回归。
- 直接变更模块优先级高于间接关联模块。
- 公共组件、公共接口、公共 SQL、公共配置影响的调用方需要分层回归。
- 历史缺陷模块、线上事故模块、强依赖上下游模块应提高回归优先级。
